Originally, bridges were treated conceptually like this:
Bridge → Core Gateway
OpenClaw's actual philosophy:
Bridge → Plugin SDK → Core Gateway
Optional capabilities should live as plugins.

OpenClaw Core should NOT understand air-gap semantics.
So tea2adt sits outside the executable trust boundary.
All encoding/decoding occurs outside OpenClaw's executable boundary.

Normally bridges are convenience layers.
With tools like tea2adt they become containment interfaces.
Therefore, tea2adt is a controlled information membrane instead of a normal transport protocol.

openclaw-tea2adt-channel-plugin:
################################
tea2adt as a plugin is one possible implementation of a "Secure / Air-Gapped Channel Plugin Specification"
it supports offline / air-gapped / asynchronous transports
- async message queues (offline synchronization)
- delayed delivery semantics
- low-bandwidth transports
- store-and-forward messaging
- human-approved message release (human-mediated channel)
- half-duplex channels
- offline threading persistence
- diode-safe interfaces
- transport capability descriptors
- policy-driven message release
- auditable message chains
